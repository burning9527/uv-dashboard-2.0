## 深圳初一暑假数据异常分析与 merged run 修复（Task #564/#565/#566/#567）
- **用户报告**：深圳初一暑假数据异常——零期有295 PV、在读PV=1299，"深圳哪来的零期，初一才700多学生怎么就1299了？"
- **根因分析**：
  1. **深圳不应有暑假零期**：深圳暑假学员无零期排课，零期PV=0是正确的。截图中的"零期=295"实际是广州数据混入。深圳寒假有12名学员有零期排课（二年级/三年级），加 season=暑假 筛选后零期 PV=0，符合预期。
  2. **深圳初一暑假在读PV应为766而非1299**：深圳初一暑假班课 PV=766（一期190+二期289+三期275+四期12），1v1 PV≈1。1299可能来自筛选器口径混乱或广州+深圳合计的一部分。
  3. **当前 merged run 全部返回0**：因 overlay_run（`20260715_205443`）数据格式为旧格式（subjects_json 缺少 year/season/grade/course_type/product_type/class_mode 等行级字段），`_merge_rows()` overlay 优先覆盖了 base 的新格式 subjects_json，导致行级筛选全部失效。
- **修复三步**：
  - **Step 1（Task #565）**：Pin `current_run_id` 到 `od_20260715_192443`（旧全量 run），临时恢复数据。
  - **Step 2（Task #566）**：根因修复——
    - `repository/__init__.py` 新增三个辅助函数：`_is_new_format_subjects(subs)` 判断格式、`_upgrade_old_subjects(subs, student_row)` 升级旧格式（从 student_row 推断 class_mode/season/grade）、`_parse_json_safe(raw)` 安全解析 JSON。重写 `_merge_rows()`：重叠 UID 时若 overlay 旧格式且 base 新格式则保留 base 的 subjects_json；校准独有 UID 时若旧格式则升级。

- **Python `__pycache__` 缓存坑**：launchd 重启进程后 Python 仍加载旧的 `.pyc` 缓存文件。每次修改 Python 源码后须执行 `find "项目路径" -type d -name __pycache__ -exec rm -rf {} +` 清除缓存，再 `launchctl kickstart -k gui/$(id -u)/com.uv.dashboard2` 重启。

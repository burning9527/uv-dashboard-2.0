## 业务趋势改为基于订单支付时间推算

### 调整
- 旧：趋势数据靠多次上传的 run 快照（每个 run = 一个数据点）
- 新：基于订单 `pay_time` 自动推算每日累计趋势（单次上传即可生成完整趋势）

### 核心逻辑
1. 遍历当前 run 的所有订单行，解析每行的 `pay_time`
2. 按 pay_time 日期排序
3. 逐日累计：截止某日 D，pay_time <= D 的订单行计入统计
4. 输出每日累计的 total_uv / active_pv / refund_pv / renewal_pv / new_count

### 修改文件
- **engine/__init__.py**:

  - 支持全维度筛选（branch/grade/season/year/course_type/class_mode/student_type/period/teaching_point/teacher/...）
- **api/__init__.py**:
  - `/api/trends/data` 端点改为调用 `compute_trends_by_paytime`（不再依赖多 run 快照）

## 业务趋势续报时间维度修复（fall_pay_time）

### 用户反馈
"为什么 7 月 4 号就到 34.2%？一期是 13 号开课的" —— 续报率在开课前就达到了开课后的最终值。
"是不是单个数据表无法抓取到这个变化" —— 用户怀疑单数据表限制。

### 根因（已解决！）
数据表**已经有**续报事件时间字段 `fall_pay_time`（续费订单的支付时间），但之前 `compute_trends_by_paytime` 用 `continue_fall=是` 这个**当前快照**判断续报，没有用 `fall_pay_time` 作为续报时间维度。

- `pay_time` = 拉新事件时间（原订单的支付时间）
- `fall_pay_time` = 续报事件时间（续费订单的支付时间，由 `enrich_cross_season_renewal` 改写）

### 修复
1. 拉新累计：按 `pay_time` 累计（拉新事件时间维度）
2. 续报累计：按 `fall_pay_time` 累计（续报事件时间维度）
3. 分离两个时间维度，避免快照 vs 事件时间错位

### 验证（一期 广州+暑假+初一+特惠+班课+一期）
| 日期 | active_pv | renewal_pv | 续报率 |
|------|-----------|------------|--------|
| 07-14 | 155 | 28 | 18.1% |
| 07-15 | 155 | 28 | 18.1% |
| 07-16 | 155 | 29 | 18.7% |
| 07-17 | 155 | 32 | 20.6% |
| 07-18 | 155 | 52 | 33.5% |
| 07-19 | 155 | 53 | 34.2% |
| 07-20 | 155 | 53 | 34.2% |

- 13 号开课次日（07-14）续报率 18.1%（开课前已续报 28 人）
- 18 号大批续报发生（28→52），19 号达到 34.2%
- 最终与 stats API 一致

### 关键设计洞察
- 单数据表确实有限制（如开课后的"调整"无法追踪）
- 但**续报事件时间已隐含在 `fall_pay_time` 字段中**——之前没发现/没利用
- 拉新和续报是两个时间维度，必须分别按 pay_time 和 fall_pay_time 累计

## 筛选器 BUG：第一排手动选择被静默清空

### 用户反馈
"刚刚，我只是把二级筛选器从一期改为零期，然后一级筛选器的筛选状态就全没有了"

### 根因
`uv-filters.js` 的 `_pruneInvalidSelections()` 对**所有 facet 字段**做剪枝（不仅是第二排联动字段）。当用户改"期次"时：
1. `_recomputeFacets()` 重新计算所有字段的 facetOptions（基于新"期次"的数据）
2. `_pruneInvalidSelections()` 检查每个字段已选值是否在新 facetOptions 中
3. 如果第一排字段的已选值不在新 facet 中 → **被静默清空** → `sel.size=0` → UI 显示"全部"（全选状态）

### 用户感知
之前用户手动选了"学科=雪球思维"等一级筛选 → 改"期次=零期"后零期没"雪球思维" → 被 prune 清空 → 显示"全部"。用户感觉"全没有了"，但实际是 size=0 = 全选。

### 修复
**第一排字段（PRIMARY_FIELDS）固定不联动**，不应被 prune 剪枝。第一排始终展示全量选项，用户的"手动选择"应该被保留——即使数据中没有这个值。
- 第一排变更：faceted 联动时只展示，但**不**强制清空
- 用户的"全选"（size=0）状态不会受影响
- 手动选择（size>0）的值在第一排保留（即使新数据没有）

### 验证
- 之前：改"期次=零期" → 第一排手动选项被清空
- 现在：改"期次=零期" → 第一排手动选项保留

## 数据洞察两图表联动失败 BUG

### 用户反馈
筛选条件下，"数据洞察" 区的"学科在读构成"和"教学点 UV 分布"显示"暂无"

### 根因
`uv-trends.js` 的 `renderInsightCharts(stats)` 渲染空态时会**替换整个 wrap**:
```js
wrap.innerHTML = UVConfig.renderEmpty({ msg: '暂无...' });
```
这导致 canvas 元素被从 DOM 中移除。下次再调用 renderInsightCharts 时:
- `doughnutEl = document.getElementById('chart-subject-doughnut')` 返回 null
- `if (doughnutEl)` 不进入 if 分支
- **永久无法再渲染图表**，即使数据有

### 修复
1. 在 if 分支加 `canvasStillInDom` 完整性检查（确认 canvas 仍是 wrap 的子元素）
2. 在 else 分支（空态）时如果 canvas 不在 DOM，**重新插入** canvas
3. 用 `display:none` 隐藏 canvas + appendChild 空态 div，而不是替换整个 wrap
4. 下次再调用时 destroy 旧 chart，重新渲染

## 图标白底填充修复

### 用户反馈
之前图标 PNG 是透明背景，macOS 显示为"带方框的白底"，不是圆角 squircle。

### 修复
用 PIL 把透明背景填充为白色（保留设计元素，填充空白区域为 #FFFFFF），让 macOS 能正确渲染：
```python
bg = Image.new("RGBA", img.size, (255, 255, 255, 255))
bg.paste(img, mask=img.split()[3])  # alpha mask
bg = bg.convert("RGB")
bg.save(OUT, "PNG")
```

### 教训
- macOS .icns 应使用**完整填充的 PNG**（没有透明），macOS 自动应用 squircle 裁剪
- 透明角导致 macOS 误判为"透明应用"，显示为方形

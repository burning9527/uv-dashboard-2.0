## 排序+卡片高度修复（第三次卡片场景化进化）

### 用户反馈
1. **"零、一、二、三这样排序，不要把零期排最后"**——零期应该排第一
2. **"切换春秋时我不希望卡片大小发生变化"**——卡片高度固定

### 根因 1：零期排序错
- `engine.compute` 内 `by_period` dict 是按数据首次出现顺序插入——非 PERIOD_ORDER 顺序
- `Flask jsonify` 会按 key 字母排序
- **双层排序错乱**——API 返回 `['一期', '二期', '零期']`（不是 0/1/2/3/4）

### 修复
1. **engine** 按 PERIOD_ORDER / WEEKDAY_ORDER 排序输出：
   ```python
   for k in sorted(by_period.keys(), key=lambda x: PERIOD_ORDER.index(x) if x in PERIOD_ORDER else 999):
       _emit(period_out, 'short', k, by_period[k])
   ```
2. **API** 自定义 `jsonify_ordered` 绕过 Flask 排序：
   ```python
   def jsonify_ordered(data):
       return Response(_json.dumps(data, ensure_ascii=False, separators=(',', ':')),
                       mimetype='application/json')
   ```
3. 替换 stats API 和 matrix API 的 `jsonify` → `jsonify_ordered`

### 根因 2：卡片高度变化
- 春秋季 `chip_display='none'` → 之前 `return ''`（容器完全消失）
- 短学期 chip 出现/消失 → 容器高度从 0 → ~40px 跳动

### 修复
1. **CSS** `.stat-card.side-hints .hint { min-height: 48px; align-content: flex-start; }` — 容器固定最小高度
2. **CSS** `.hint.hint-empty` — 空状态灰色斜体"暂无"占位
3. **JS** 长学期返回 `'<div class="hint hint-empty">长学期（春/秋）</div>'`（保留容器）
4. **JS** 短学期但无数据时返回 `'<div class="hint hint-empty">暂无期次数据</div>'`

### 验证排序
- 寒+暑 by_period: `['零期', '一期', '二期', '三期', '四期']` ✓
- 秋季 by_weekday: `['周四', '周五', '周六', '周日']` ✓
- 春秋季：hint-empty 占位 → 卡片大小保持 ✓

筛选 深圳+26暑假+初一，渠道图总数应等于 810，但实际只有 244（仅 3 渠道），缺社群/商务。

### 根因
1. by_channel 只统计 `_st_main == '新生'`（新生学员）
2. "社群" 和 "商务" 渠道只有 1 学员（老生），所以 acquired=0，被 `if acquired <= 0: continue` 过滤掉
3. 用户期望：教学点统计 = **所有通过筛选的学员**（不只新生）—— 匹配 Excel "5 项 810 条"

### 修复
**engine/__init__.py:645-679** — by_channel 计算改为：
- **所有通过筛选的学员**按 channel_l1/l2 分类（不只是新生）
- 字段：
  - `total_uv`: 该渠道所有学员数（匹配 Excel 教学点）
  - `acquired`: 新生学员数
  - `active_new`: 在读新生
  - `renewed`: 已续
  - `refunded`: 退费

**api/__init__.py:499-509** — 改为按 `total_uv > 0` 显示（而不是 `acquired > 0`）：
- 商务/社群即使 acquired=0 也会显示（因为 total_uv=1）
- 返回 total_uv 字段给前端

  - 其他: acquired=103, total_uv=168

## student_type 筛选口径 BUG 修复

### 用户反馈
筛选 深圳+F列2026+G列暑假+I列一期+K列特惠+P列初一+BS列拉新，期望 active_pv=125，但显示 54。

### 根因
**student_type 筛选使用"学员级聚合 + 优先级去重"**：
- 某学员有一期"新生"行 + 三期"老生"行（续报后变老生）
- 学员级聚合 _st_main = 老生（高优先级）
- 筛选 student_type=新生 → 学员级"老生"不在 {"新生"} → continue 跳过整个学员

但 Excel BS 列是**行级**——只看每行的 student_type 字段。Excel 125 行，新生行级筛 125 行；DB 学员级筛只匹配 41 学员。

### 修复
**engine/__init__.py**:
1. 学员级筛改为"行级优先"：学员至少有一行 student_type 在 fs.student_type → 纳入
   ```python
   if not (_st_set & set(fs.student_type)):
       continue
   ```
2. 行级 filter 添加 student_type 检查（只算匹配的行）
   ```python
   if fs.student_type:
       line_st = (sd.get('student_type') or '').strip()
       if line_st and line_st not in set(fs.student_type):
           continue
   ```

### 验证
| 指标 | 修复前 | 修复后 | Excel 期望 |
|------|--------|--------|------------|
| total_uv | 41 | 85 | ~ |
| active_uv | 39 | 83 | ~ |
| **active_pv** | **54** | **120** | **125** |
| refund_pv | 4 | 5 | ~ |

### 测试更新
**test_filters_v3.py** "学员归属加和 vs 全局" 测试需要更新：
- 旧版"学员级聚合 + 优先级去重"：加和 = 全局
- 新版"行级筛"：加和 ≥ 全局（跨 set 重复计数是预期行为）
- 测试改为 `sum_st >= ALL_ACTIVE * 0.95`（行级筛加和只会 ≥ 全局）

## 拉新渠道图堆叠 vs active_pv 数量对不上

### 用户反馈
顶部数据卡 active_pv=122，渠道图堆叠 active_new 总和不是 122。

### 根因（三层 BUG）
1. **行级 channel_l1/ch2 缺失**：subjects_json 行级 dict 没有 channel_l1/channel_l2 字段（loader 之前未加），导致 by_channel 行级累加时全部为空 → 累计到 0
2. **引擎用学员级 channel_l1**（单一值），不是行级 → 同一学员多渠道行都算学员级 ch1
3. **ACQ_CHAN_FIELDS 不含 student_type** → API 端点 `fs_kwargs` 没传 student_type → fs.student_type=[] 不过滤 → 渠道图算的是"深圳+暑+一期+特惠+初一"（不限归属）所有学员，不只是新生

### 修复
1. **`engine/order_detail_loader.py` _build_line**: 增加 `channel_l1`/`channel_l2` 行级字段（来自 `row.channel_lead_l1/l2`）
2. **`engine/order_detail_loader.py`** + **`repository/_upgrade_old_subjects`**: 行级 None 时补学员级 channel_l1/l2 fallback
3. **`engine/__init__.py` by_channel 累加**: 
   - 在 is_active 块中按行级 `sd.channel_l1/l2` 累加 active_new/renewed
   - 在 is_refund 块中按行级累加 refunded
   - 学员级循环只保留 acquired/total_uv 学员级去重
4. **`api/__init__.py` ACQ_CHAN_FIELDS**: 增加 'student_type'，使 API 把 student_type 传给 FilterSpec
5. **手动 SQL 迁移**: 新 merged run `merged_20260717_191114` 中"新宝利"/"天成"→ 标准化名（每次 rebuild 都会出现，需定期执行）
